# Sponsors API Specification

Base URL: `{API_BASE_URI}` (configured via environment).

All endpoints expect/return `application/json`. Routes are reachable both with and without the `/api` prefix unless explicitly noted (e.g. `/sponsors` and `/api/sponsors` are equivalent).

This document covers the full Sponsors surface — list / create / delete plus the per-resource upload alias. The media-storage / presign contract used by `POST /sponsors/upload` is documented once in [`MEDIA_API.md`](./MEDIA_API.md) and is shared with the other per-resource upload aliases.

Sponsors are intentionally the leanest resource in the API: they have no read-by-id, no update, no status, and no sport — just a name and a logo. The frontend renders them as a marquee / footer strip, so the only operations the admin needs are "list all", "add new", and "remove one".

---

## Concepts

### `id` (not `slug`)

Unlike most resources, sponsors are addressed by a numeric / opaque `id` — the `DELETE` route is `DELETE /sponsors/:id`, not `:slug`. The backend MUST return `id` on every `GET /sponsors` item and on the `POST /sponsors` response.

### No `GET /sponsors/:id` and no `PUT`

There is intentionally no fetch-by-id and no update endpoint. To change a sponsor's name or logo, the admin deletes it and creates a new one. This matches the current frontend, which only ever lists or destroys.

### Authentication

- `GET / POST / DELETE` routes are **not** protected in code today, matching the rest of the resource spec.
- `POST /sponsors/upload` IS protected — send `x-access-token: <JWT>`. This mirrors all other `/<resource>/upload` aliases.

## Schema

### `Sponsor`

```jsonc
{
  "id":        "string | number",                                                   // canonical id, generated by backend
  "name":      "string",
  "logoUrl":   "string | null",                                                     // URL returned from /sponsors/upload (CDN / public URL)
  "createdAt": "ISO8601",
  "updatedAt": "ISO8601"
}
```

---

## Endpoints

### `GET /sponsors`

```jsonc
Response 200: Sponsor[]
```

### `POST /sponsors`

```jsonc
Request:
{
  "name":    "string",                                                              // required
  "logoUrl": "string?"                                                              // set this to the `url` returned by /sponsors/upload
}

Response 201: Sponsor   // { id, name, logoUrl, ... }
```

### `DELETE /sponsors/:id`

```jsonc
Response 204
Response 404: { "error": "Sponsor not found" }
```

## Logo upload

### `POST /sponsors/upload`

Protected. Returns presigned S3 upload details. Same response shape as `POST /media/presign` with the folder hard-coded to `sponsors` — the client SHOULD omit `folder` when calling the alias; the backend MUST ignore any client-supplied `folder` value and use `sponsors`.

See [`MEDIA_API.md`](./MEDIA_API.md) for:
- the canonical request / response payload,
- the browser → S3 PUT flow (the API server never sees the file bytes),
- backend signing responsibilities (auth, key generation, size + content-type pinning, short expiry),
- and S3 bucket / CORS requirements.

Once S3 returns `2xx` on the `PUT`, the client stores the returned `url` in `Sponsor.logoUrl` and submits the `POST /sponsors` as usual.

## Error format

All non-2xx responses use the project-wide error shape:

```json
{ "error": "human-readable message", "code": "OPTIONAL_MACHINE_CODE" }
```
