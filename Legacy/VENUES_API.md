# Venues API Specification

Base URL: `{API_BASE_URI}` (configured via environment).

All endpoints expect/return `application/json`. Routes are reachable both with and without the `/api` prefix unless explicitly noted (e.g. `/venues` and `/api/venues` are equivalent).

This document covers the full Venues surface â€” CRUD plus the per-resource upload alias. The media-storage / presign contract used by `POST /venues/upload` is documented once in [`MEDIA_API.md`](./MEDIA_API.md) and is shared with the other per-resource upload aliases.

## Concepts

### Slug

Every venue is identified by a URL-friendly `slug` (not a numeric id). The backend generates the slug, likely from `name`. The slug MUST be returned on every GET response and is used as the path parameter on `GET /venues/:slug`, `PUT /venues/:slug`, and `DELETE /venues/:slug`.

### Enums

| Field      | Values                                                                                                                                  | Default     |
|------------|-----------------------------------------------------------------------------------------------------------------------------------------|-------------|
| `type`     | `STADIUM \| ARENA \| GROUND \| INDOOR_COURT \| OUTDOOR_COURT \| CLUBHOUSE \| OTHER`                                                      | required    |
| `status`   | `ACTIVE \| INACTIVE \| CLOSED`                                                                                                          | `ACTIVE`    |
| `emirate`  | `ABU_DHABI \| DUBAI \| SHARJAH \| AJMAN \| UMM_AL_QUWAIN \| RAS_AL_KHAIMAH \| FUJAIRAH \| null`                                          | nullable    |

The `emirate` enum is UAE-specific and nullable for venues outside the UAE; in that case `country` / `city` carry the location.

### Relationship to other resources

- **Matches** (see [`MATCHES_API.md`](./MATCHES_API.md)) and **Other Events** (see [`OTHER_EVENTS_API.md`](./OTHER_EVENTS_API.md)) both reference a venue via a free-text `venue` string (e.g. `"Dubai International Cricket Stadium"`), **not** a FK to `Venue.slug`. The backend MAY reconcile those strings against existing `Venue` records but is not required to enforce the FK.

Renaming a `Venue` therefore does NOT cascade into existing matches or events. If a rename happens, the backend SHOULD update referencing rows out-of-band.

### Authentication

- CRUD routes (`GET / POST / PUT / DELETE`) are **not** protected in code today, matching the rest of the resource spec.
- `POST /venues/upload` IS protected â€” send `x-access-token: <JWT>`. This mirrors all other `/<resource>/upload` aliases.

## Schema

### `Venue`

```jsonc
{
  "slug":        "string",                                                          // canonical id, generated by backend
  "name":        "string",
  "type":        "STADIUM | ARENA | GROUND | INDOOR_COURT | OUTDOOR_COURT | CLUBHOUSE | OTHER",
  "status":      "ACTIVE | INACTIVE | CLOSED",
  "description": "string | null",
  "capacity":    "number | null",
  "address":     "string | null",
  "emirate":     "ABU_DHABI | DUBAI | SHARJAH | AJMAN | UMM_AL_QUWAIN | RAS_AL_KHAIMAH | FUJAIRAH | null",
  "city":        "string | null",
  "country":     "string | null",
  "website":     "string | null",
  "email":       "string | null",
  "phone":       "string | null",
  "imageUrl":    "string | null",                                                   // URL returned from /venues/upload (CDN / public URL)
  "createdAt":   "ISO8601",
  "updatedAt":   "ISO8601"
}
```

## Endpoints

### `GET /venues`

```jsonc
Response 200: Venue[]
```

### `GET /venues/:slug`

```jsonc
Response 200: Venue
Response 404: { "error": "Venue not found" }
```

### `POST /venues`

```jsonc
Request:
{
  "name":        "string",                                                          // required
  "type":        "STADIUM | ARENA | GROUND | INDOOR_COURT | OUTDOOR_COURT | CLUBHOUSE | OTHER", // required
  "status":      "ACTIVE | INACTIVE | CLOSED",                                      // required, default ACTIVE
  "description": "string?",
  "capacity":    "number?",
  "address":     "string?",
  "emirate":     "ABU_DHABI | DUBAI | SHARJAH | AJMAN | UMM_AL_QUWAIN | RAS_AL_KHAIMAH | FUJAIRAH | null",
  "city":        "string?",
  "country":     "string?",
  "website":     "string?",
  "email":       "string?",
  "phone":       "string?",
  "imageUrl":    "string?"                                                          // set this to the `url` returned by /venues/upload
}

Response 201: Venue
```

### `PUT /venues/:slug`

Same request shape as `POST`. Treat as a full replacement of the venue document â€” the client always sends the complete state.

```jsonc
Response 200: Venue
Response 404: { "error": "Venue not found" }
```

### `DELETE /venues/:slug`

```jsonc
Response 204
```

## Image upload

### `POST /venues/upload`

Protected. Returns presigned S3 upload details. Same response shape as `POST /media/presign` with the folder hard-coded to `venues` â€” the client SHOULD omit `folder` when calling the alias; the backend MUST ignore any client-supplied `folder` value and use `venues`.

See [`MEDIA_API.md`](./MEDIA_API.md) for:
- the canonical request / response payload,
- the browser â†’ S3 PUT flow (the API server never sees the file bytes),
- backend signing responsibilities (auth, key generation, size + content-type pinning, short expiry),
- and S3 bucket / CORS requirements.

Once S3 returns `2xx` on the `PUT`, the client stores the returned `url` in `Venue.imageUrl` and submits the `POST /venues` or `PUT /venues/:slug` as usual.

## Error format

All non-2xx responses use the project-wide error shape:
