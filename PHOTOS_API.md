# Photos API Specification

Base URL: `{API_BASE_URI}` (configured via environment).

All endpoints expect/return `application/json`. Routes are reachable both with and without the `/api` prefix unless explicitly noted (e.g. `/photos` and `/api/photos` are equivalent).

This document covers the full Photos surface — CRUD plus the per-resource upload alias. The media-storage / presign contract used by `POST /photos/upload` is documented once in [`MEDIA_API.md`](./MEDIA_API.md) and is shared with the other per-resource upload aliases.

"Photos" are entries in the public photo gallery — each one is a single image with a title, optional description, and a publish/draft flag. They are **not** related to `Player.photoUrl` (a player headshot) or any other `*Url` field on other resources; those reuse the same S3 contract but are not entries in this collection.

---

## Concepts

### Slug

Every photo is identified by a URL-friendly `slug` (not a numeric id). The backend generates the slug, likely from `title`. The slug MUST be returned on every GET response and is used as the path parameter on `GET /photos/:slug`, `PUT /photos/:slug`, and `DELETE /photos/:slug`.

### Enums

| Field    | Values                | Default   |
|----------|-----------------------|-----------|
| `status` | `PUBLISHED \| DRAFT`  | `DRAFT`   |

`DRAFT` photos MUST be excluded from any public-facing list. They remain reachable by `slug` for admin preview.

### Authentication

- CRUD routes (`GET / POST / PUT / DELETE`) are **not** protected in code today, matching the rest of the resource spec.
- `POST /photos/upload` IS protected — send `x-access-token: <JWT>`. This mirrors all other `/<resource>/upload` aliases.

## Schema

### `Photo`

```jsonc
{
  "slug":        "string",                                                          // canonical id, generated by backend
  "title":       "string",
  "status":      "PUBLISHED | DRAFT",
  "description": "string | null",
  "imageUrl":    "string | null",                                                   // URL returned from /photos/upload (CDN / public URL)
  "createdAt":   "ISO8601",
  "updatedAt":   "ISO8601"
}
```

## Endpoints

### `GET /photos`

```jsonc
Response 200: Photo[]
```

### `GET /photos/:slug`

```jsonc
Response 200: Photo
Response 404: { "error": "Photo not found" }
```

### `POST /photos`

```jsonc
Request:
{
  "title":       "string",                                                          // required
  "status":      "PUBLISHED | DRAFT",                                               // required, default DRAFT
  "description": "string?",
  "imageUrl":    "string?"                                                          // set this to the `url` returned by /photos/upload
}

Response 201: Photo
```

### `PUT /photos/:slug`

Same request shape as `POST`. Treat as a full replacement of the photo document — the client always sends the complete state.

```jsonc
Response 200: Photo
Response 404: { "error": "Photo not found" }
```

### `DELETE /photos/:slug`

```jsonc
Response 204
```

## Image upload

### `POST /photos/upload`

Protected. Returns presigned S3 upload details. Same response shape as `POST /media/presign` with the folder hard-coded to `photos` — the client SHOULD omit `folder` when calling the alias; the backend MUST ignore any client-supplied `folder` value and use `photos`.

See [`MEDIA_API.md`](./MEDIA_API.md) for:
- the canonical request / response payload,
- the browser → S3 PUT flow (the API server never sees the file bytes),
- backend signing responsibilities (auth, key generation, size + content-type pinning, short expiry),
- and S3 bucket / CORS requirements.

Once S3 returns `2xx` on the `PUT`, the client stores the returned `url` in `Photo.imageUrl` and submits the `POST /photos` or `PUT /photos/:slug` as usual.

---

## Error format

All non-2xx responses use the project-wide error shape:

```json
{ "error": "human-readable message", "code": "OPTIONAL_MACHINE_CODE" }
```
