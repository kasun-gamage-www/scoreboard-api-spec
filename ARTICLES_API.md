# Articles API Specification

Base URL: `{API_BASE_URI}` (configured via environment).

All endpoints expect/return `application/json`. Routes are reachable both with and without the `/api` prefix unless explicitly noted (e.g. `/articles` and `/api/articles` are equivalent).

This document covers the full Articles surface — CRUD plus the per-resource upload alias. The media-storage / presign contract used by `POST /articles/upload` is documented once in [`MEDIA_API.md`](./MEDIA_API.md) and is shared with the other per-resource upload aliases (`/clubs/upload`, `/photos/upload`, etc.).

---

## Concepts

### Slug

Every article is identified by a URL-friendly `slug` (not a numeric id). The backend generates the slug, likely from `title`. The slug MUST be returned on every GET response and is used as the path parameter on `GET /articles/:slug`, `PUT /articles/:slug`, and `DELETE /articles/:slug`.

### `category`

`NEWS | MATCH_REPORT | FEATURE | INTERVIEW | OPINION`

### `sport`

`CRICKET | RUGBY | SOCCER | BADMINTON | BOWLING | MOTOR_SPORTS | BASKETBALL | NETBALL | VOLLEYBALL | CYCLING`

### `status`

| Status      | Visible publicly? | Notes                                   |
|-------------|-------------------|-----------------------------------------|
| `DRAFT`     | no                | Default for new articles. Admin only.   |
| `PUBLISHED` | yes               | Listed and fetchable by readers.        |
| `ARCHIVED`  | no                | Removed from public lists but retained. |

`DRAFT` and `ARCHIVED` articles MUST be excluded from any public-facing list. They remain reachable by `slug` for admin preview.

### Authentication

- CRUD routes (`GET / POST / PUT / DELETE`) are **not** protected in code today, matching the rest of the resource spec.
- `POST /articles/upload` IS protected — send `x-access-token: <JWT>`. This mirrors all other `/<resource>/upload` aliases.

## Schema

### `Article`

```jsonc
{
  "slug":            "string",                                                       // canonical id, generated by backend
  "title":           "string",
  "author":          "string",
  "content":         "string (html)",
  "category":        "NEWS | MATCH_REPORT | FEATURE | INTERVIEW | OPINION",
  "sport":           "CRICKET | RUGBY | SOCCER | BADMINTON | BOWLING | MOTOR_SPORTS | BASKETBALL | NETBALL | VOLLEYBALL | CYCLING",
  "status":          "PUBLISHED | DRAFT | ARCHIVED",
  "summary":         "string | null",
  "imageUrl":        "string | null",        // URL returned from /articles/upload (CDN / public URL)
  "imageYAlignment": "top | center | bottom | custom | null", // hero crop hint for `imageUrl`; default `center`
  "imageYOffset":    "number | null",        // pixel offset from top, set only when imageYAlignment === "custom"; otherwise null
  "createdAt":       "ISO8601",
  "updatedAt":       "ISO8601"
}
```

`imageYAlignment` / `imageYOffset` are presentation hints used by the public site when rendering the hero image. They are stored and round-tripped by the backend but have no effect on validation beyond the enum / numeric type checks. When `imageYAlignment` is anything other than `custom`, `imageYOffset` MUST be `null`.

## Endpoints

### `GET /articles`

```jsonc
Response 200: Article[]
```

### `GET /articles/:slug`

```jsonc
Response 200: Article
Response 404: { "error": "Article not found" }
```

### `POST /articles`

```jsonc
Request:
{
  "title":           "string",                                                          // required
  "author":          "string",                                                          // required
  "content":         "string (html)",                                                   // required
  "category":        "NEWS | MATCH_REPORT | FEATURE | INTERVIEW | OPINION",             // required
  "sport":           "CRICKET | RUGBY | SOCCER | BADMINTON | BOWLING | MOTOR_SPORTS | BASKETBALL | NETBALL | VOLLEYBALL | CYCLING", // required
  "status":          "PUBLISHED | DRAFT | ARCHIVED",                                    // required, default DRAFT
  "summary":         "string?",
  "imageUrl":        "string?",                                                         // set this to the `url` returned by /articles/upload
  "imageYAlignment": "top | center | bottom | custom",                                  // optional, defaults to `center`
  "imageYOffset":    "number?"                                                          // required when imageYAlignment === "custom"; MUST be null otherwise
}

Response 201: Article
```

### `PUT /articles/:slug`

Same request shape as `POST`. Treat as a full replacement of the article document — the client always sends the complete state.

```jsonc
Response 200: Article
Response 404: { "error": "Article not found" }
```

### `DELETE /articles/:slug`

```jsonc
Response 204
```

## Image upload

### `POST /articles/upload`

Protected. Returns presigned S3 upload details. Same response shape as `POST /media/presign` with the folder hard-coded to `articles` — the client SHOULD omit `folder` when calling the alias; the backend MUST ignore any client-supplied `folder` value and use `articles`.

See [`MEDIA_API.md`](./MEDIA_API.md) for:
- the canonical request / response payload,
- the browser → S3 PUT flow (the API server never sees the file bytes),
- backend signing responsibilities (auth, key generation, size + content-type pinning, short expiry),
- and S3 bucket / CORS requirements.

Once S3 returns `2xx` on the `PUT`, the client stores the returned `url` in `Article.imageUrl` and submits the `POST /articles` or `PUT /articles/:slug` as usual.

---

## Error format

All non-2xx responses use the project-wide error shape:

```json
{ "error": "human-readable message", "code": "OPTIONAL_MACHINE_CODE" }
```
