# Players API Specification

Base URL: `{API_BASE_URI}` (configured via environment).

All endpoints expect/return `application/json`. Routes are reachable both with and without the `/api` prefix unless explicitly noted (e.g. `/players` and `/api/players` are equivalent).

This document covers the full Players surface ‚Äî CRUD plus the per-resource upload alias. The media-storage / presign contract used by `POST /players/upload` is documented once in [`MEDIA_API.md`](./MEDIA_API.md) and is shared with the other per-resource upload aliases.

---

## Concepts

### Slug

Every player is identified by a URL-friendly `slug` (not a numeric id). The backend generates the slug, likely from `name`. The slug MUST be returned on every GET response and is used as the path parameter on `GET /players/:slug`, `PUT /players/:slug`, and `DELETE /players/:slug`.

### Enums

| Field    | Values                                                                                                                                 | Default      |
|----------|----------------------------------------------------------------------------------------------------------------------------------------|--------------|
| `sport`  | `CRICKET \| RUGBY \| SOCCER \| BADMINTON \| BOWLING \| MOTOR_SPORTS \| BASKETBALL \| NETBALL \| VOLLEYBALL \| CYCLING \| OTHER`         | required     |
| `status` | `ACTIVE \| INACTIVE`                                                                                                                   | `ACTIVE`     |

### Inline creation from the match form

Players are also created indirectly when an admin types a player name into a match's `homeTeamPlayers` / `awayTeamPlayers` list (see [`MATCHES_API.md`](./MATCHES_API.md)). When that happens the embedded entry carries a `jerseyNumber` and an optional `position`; the backend SHOULD reconcile against / promote to a real `Player` record (matching by name within the team's sport) and set `playerId` on the embedded entry. `jerseyNumber` and `position` on the `Player` record reflect the most recent inline submission.

### Authentication

- CRUD routes (`GET / POST / PUT / DELETE`) are **not** protected in code today, matching the rest of the resource spec.
- `POST /players/upload` IS protected ‚Äî send `x-access-token: <JWT>`. This mirrors all other `/<resource>/upload` aliases.

## Schema

### `Player`

```jsonc
{
  "slug":         "string",                                                       // canonical id, generated by backend
  "name":         "string",
  "sport":        "CRICKET | RUGBY | SOCCER | BADMINTON | BOWLING | MOTOR_SPORTS | BASKETBALL | NETBALL | VOLLEYBALL | CYCLING | OTHER",
  "status":       "ACTIVE | INACTIVE",
  "bio":          "string | null",
  "team":         "string | null",                                                // free-text team name; not a Team.slug FK
  "position":     "string | null",
  "country":      "string | null",
  "photoUrl":     "string | null",                                                // URL returned from /players/upload (CDN / public URL)
  "jerseyNumber": "number | null",                                                // last value seen on a match's homeTeamPlayers / awayTeamPlayers entry
  "createdAt":    "ISO8601",
  "updatedAt":    "ISO8601"
}
```

## Endpoints

### `GET /players`

```
Query params:
  search?: string   // case-insensitive filter on `name` (substring match)

Response 200: Player[]
```

### `GET /players/:slug`

```jsonc
Response 200: Player
Response 404: { "error": "Player not found" }
```

### `POST /players`

```jsonc
Request:
{
  "name":         "string",                                                       // required
  "sport":        "CRICKET | RUGBY | SOCCER | BADMINTON | BOWLING | MOTOR_SPORTS | BASKETBALL | NETBALL | VOLLEYBALL | CYCLING | OTHER", // required
  "status":       "ACTIVE | INACTIVE",                                            // required, default ACTIVE
  "bio":          "string?",
  "team":         "string?",
  "position":     "string?",
  "country":      "string?",
  "photoUrl":     "string?",                                                      // set this to the `url` returned by /players/upload
  "jerseyNumber": "number?"                                                       // used when added inline from match form
}

Response 201: Player
```

### `PUT /players/:slug`

Same request shape as `POST`. Treat as a full replacement of the player document ‚Äî the client always sends the complete state.

```jsonc
Response 200: Player
Response 404: { "error": "Player not found" }
```

### `DELETE /players/:slug`

```jsonc
Response 204
```

## Photo upload

### `POST /players/upload`

Protected. Returns presigned S3 upload details. Same response shape as `POST /media/presign` with the folder hard-coded to `players` ‚Äî the client SHOULD omit `folder` when calling the alias; the backend MUST ignore any client-supplied `folder` value and use `players`.

See [`MEDIA_API.md`](./MEDIA_API.md) for:
- the canonical request / response payload,
- the browser ‚Üí S3 PUT flow (the API server never sees the file bytes),
- backend signing responsibilities (auth, key generation, size + content-type pinning, short expiry),
- and S3 bucket / CORS requirements.

Once S3 returns `2xx` on the `PUT`, the client stores the returned `url` in `Player.photoUrl` and submits the `POST /players` or `PUT /players/:slug` as usual.

---

## Error format

All non-2xx responses use the project-wide error shape:

```json
{ "error": "human-readable message", "code": "OPTIONAL_MACHINE_CODE" }
```
