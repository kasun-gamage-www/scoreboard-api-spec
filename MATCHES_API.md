# Matches API Specification

Base URL: `{API_BASE_URI}` (configured via environment).

All endpoints expect/return `application/json`. Routes are reachable both with and without the `/api` prefix (e.g. `/matches` and `/api/matches` are equivalent).

This document covers the full `/matches` surface: CRUD, the featured-match flag, and the bulk CSV / Excel schedule importer. Cricket-specific use of this resource (the score editor) is documented in [`CRICKET_API.md`](./CRICKET_API.md) → §2 Admin score editor.

---

## Concepts

### Slug

Every match is identified by a URL-friendly `slug` (not a numeric id). The backend generates the slug, likely from `<homeTeamName>-vs-<awayTeamName>` + a disambiguator. The slug MUST be returned on every GET response and is used as the path parameter on `GET /matches/:slug`, `PUT /matches/:slug`, `DELETE /matches/:slug`, `PUT /matches/:slug/featured`, and `DELETE /matches/:slug/featured`.

### Enums

| Field       | Values                                                                                                                          | Default       |
|-------------|---------------------------------------------------------------------------------------------------------------------------------|---------------|
| `sport`     | `CRICKET \| RUGBY \| SOCCER \| BADMINTON \| BOWLING \| MOTOR_SPORTS \| BASKETBALL \| NETBALL \| VOLLEYBALL \| CYCLING \| GOLF`   | required      |
| `status`    | `SCHEDULED \| LIVE \| COMPLETED \| CANCELLED \| POSTPONED`                                                                      | `SCHEDULED`   |
| `matchType` | `LEAGUE \| KNOCKOUT \| FRIENDLY \| INDEPENDENT \| null`                                                                          | `null`        |

Note: the match `sport` enum does NOT include `OTHER` (which exists on `Player.sport`, `Team.sport`, etc.). A match is always associated with a recognised competitive sport.

### Teams and players (free-text, not FKs)

`homeTeamName` and `awayTeamName` are free-text strings, not foreign keys to `Team.slug`. Same for the per-player `name` inside `homeTeamPlayers` / `awayTeamPlayers`. See [`TEAMS_API.md`](./TEAMS_API.md) and [`PLAYERS_API.md`](./PLAYERS_API.md) for how the backend MAY reconcile these names against existing records.

For the embedded player entries, `playerId` is the optional FK to a `Player`; the backend SHOULD set it when reconciling an inline-added player. `jerseyNumber` and `position` on the embedded entry are scoped to this match — they don't have to match the canonical `Player` record.

### Sport-specific `details`

The `details` object carries sport-specific scoring fields. Each sport uses a disjoint set of keys (see [Schema](#schema) below). The backend MUST:

- **Preserve unknown keys verbatim** on every write. The cricket editor, for example, only patches cricket keys and echoes everything else back — football / rugby / etc. keys that may already be on the document MUST NOT be dropped.
- Treat `PUT` as a **full replacement** of the document the client sends, NOT a deep merge. The client always sends the complete state it wants persisted.

### Cricket format inherited from tournament

On `POST /matches` and `PUT /matches/:slug`, if the request has a non-null `tournamentSlug` and `details.cricketFormat` is missing or `null`, the backend MUST copy the linked tournament's `cricketFormat` onto the stored match. The inherited value is persisted on the match document — subsequent reads are self-contained and do not re-resolve against the tournament. A non-null `details.cricketFormat` in the request always wins over the tournament value. See [`TOURNAMENTS_API.md`](./TOURNAMENTS_API.md) → Cricket format inheritance.

### Featured flag (at-most-one invariant)

Every `Match` carries `featured: boolean`. The invariant the backend MUST enforce is: **at most one match has `featured: true` at any time across the whole table**.

- `featured` is read-only on `POST /matches` and `PUT /matches/:slug` — the client MUST NOT set it via the generic CRUD endpoints; the backend MUST ignore the value and default to `false` on create.
- Use `PUT /matches/:slug/featured` to mark a match featured. The backend MUST atomically unset `featured` on the previously featured match in the same transaction.
- Use `DELETE /matches/:slug/featured` to clear the flag (no-op if it wasn't featured).

### Authentication

- CRUD routes (`GET / POST / PUT / DELETE`) and the featured-flag routes are **not** protected in code today, matching the rest of the resource spec.
- `POST /matches/upload-schedule` IS protected — send `x-access-token: <JWT>`.

## Schema

### `Match`

```jsonc
{
  "slug":           "string",                                                       // canonical id, generated by backend
  "sport":          "CRICKET | RUGBY | SOCCER | BADMINTON | BOWLING | MOTOR_SPORTS | BASKETBALL | NETBALL | VOLLEYBALL | CYCLING | GOLF",
  "homeTeamName":   "string",                                                       // free-text team name
  "awayTeamName":   "string",
  "status":         "SCHEDULED | LIVE | COMPLETED | CANCELLED | POSTPONED",
  "tournamentSlug": "string | null",                                                // FK to Tournament.slug; null for independent / friendly matches
  "matchType":      "LEAGUE | KNOCKOUT | FRIENDLY | INDEPENDENT | null",
  "venue":          "string | null",
  "scheduledAt":    "ISO8601 | null",
  "referee":        "string | null",
  "notes":          "string | null",                                                // free-text result / commentary line
  "homeScore":      "number | null",                                                // match-total override; admin-managed, not derived
  "awayScore":      "number | null",
  "featured":       "boolean",                                                      // read-only via CRUD; set via /matches/:slug/featured

  "details": SportDetails,                                                          // see below — sport-specific scoring keys

  "homeTeamPlayers": [
    { "name": "string", "jerseyNumber": "number?", "position": "string?", "playerId": "string | null" }
  ],
  "awayTeamPlayers": [
    { "name": "string", "jerseyNumber": "number?", "position": "string?", "playerId": "string | null" }
  ],

  "createdAt":      "ISO8601",
  "updatedAt":      "ISO8601"
}
```

### `SportDetails`

All keys are optional. Different sports use disjoint subsets; keys from sports other than the current `match.sport` MAY be present (left over from a sport change) and MUST be preserved on write.

```jsonc
{
  // ---- Cricket ----
  // Innings format: runs (≥0), wickets (0..10), overs (X.Y with Y in 0..5 — 4.6 is illegal, balls roll over to 5.0).
  // `cricketFormat` discriminates the game length: T20 and OD use Inn1 only per side; TEST may use both.
  // When the match has a `tournamentSlug`, a missing/null `cricketFormat` is inherited from the tournament on write.
  // Cricket editor details: see CRICKET_API.md §2.
  "cricketFormat":    "T20 | OD | TEST | null",
  "homeInn1Runs":     "number?", "homeInn1Wickets": "number?", "homeInn1Overs": "number?",
  "homeInn2Runs":     "number?", "homeInn2Wickets": "number?", "homeInn2Overs": "number?",
  "awayInn1Runs":     "number?", "awayInn1Wickets": "number?", "awayInn1Overs": "number?",
  "awayInn2Runs":     "number?", "awayInn2Wickets": "number?", "awayInn2Overs": "number?",

  // ---- Football (Soccer) ----
  "extraTime":     "boolean?",
  "homeGoalsET":   "number?", "awayGoalsET":   "number?",
  "penalties":     "boolean?",
  "homePenalties": "number?", "awayPenalties": "number?",

  // ---- Rugby ----
  "homeTries":     "number?", "homeConversions": "number?", "homeDropGoals": "number?", "homeRugbyPenalties": "number?",
  "awayTries":     "number?", "awayConversions": "number?", "awayDropGoals": "number?", "awayRugbyPenalties": "number?",

  // ---- Basketball (per quarter, with optional overtime) ----
  "homeQ1": "number?", "homeQ2": "number?", "homeQ3": "number?", "homeQ4": "number?", "homeOT": "number?",
  "awayQ1": "number?", "awayQ2": "number?", "awayQ3": "number?", "awayQ4": "number?", "awayOT": "number?",

  // ---- Tennis (array, max 5 sets) ----
  "sets": [{ "homeGames": "number", "awayGames": "number" }],

  // ---- Golf ----
  // Format discriminates the rest of the golf block. See SPORTS/GOLF_API.md §2 for the full shapes.
  "format":          "STROKE_PLAY? | MATCH_PLAY? | TEAM_COMPETITION?",
  "coursePar":       "number?", "courseYards": "number?",
  "rounds":          "number?", "currentRound": "number?",
  "cutLine":         "number | null",
  "winner":          "string?",
  "leaderboard":     "LeaderboardEntry[]?",        // STROKE_PLAY (and team-aggregate)
  "matchPlay":       "MatchPlayCard?",             // MATCH_PLAY
  "teamCompetition": "TeamCompetitionDetail?"      // TEAM_COMPETITION
}
```

Golf-specific use of this resource (the score editor) is documented in [`SPORTS/GOLF_API.md`](./SPORTS/GOLF_API.md) → §2 Admin score editor.

## CRUD endpoints

### `GET /matches`

```jsonc
Response 200: Match[]                                                             // each Match includes `featured: boolean`; at most one is true
```

### `GET /matches/:slug`

```jsonc
Response 200: Match
Response 404: { "error": "Match not found" }
```

### `POST /matches`

```jsonc
Request:
{
  "sport":          "CRICKET | RUGBY | SOCCER | BADMINTON | BOWLING | MOTOR_SPORTS | BASKETBALL | NETBALL | VOLLEYBALL | CYCLING | GOLF", // required
  "homeTeamName":   "string",                                                     // required
  "awayTeamName":   "string",                                                     // required
  "status":         "SCHEDULED | LIVE | COMPLETED | CANCELLED | POSTPONED",       // required, default SCHEDULED
  "tournamentSlug": "string | null",
  "matchType":      "LEAGUE | KNOCKOUT | FRIENDLY | INDEPENDENT | null",
  "venue":          "string?",
  "scheduledAt":    "ISO8601?",
  "referee":        "string?",
  "notes":          "string?",
  "homeScore":      "number | null",
  "awayScore":      "number | null",
  "featured":       "boolean (IGNORED on POST/PUT — set via /matches/:slug/featured; defaults to false)",
  "details":        SportDetails,
  "homeTeamPlayers": [
    { "name": "string", "jerseyNumber": "number?", "position": "string?", "playerId": "string | null" }
  ],
  "awayTeamPlayers": [
    { "name": "string", "jerseyNumber": "number?", "position": "string?", "playerId": "string | null" }
  ]
}

Response 201: Match
```

### `PUT /matches/:slug`

Same request shape as `POST`. Treat as a full replacement of the match document — the client always sends the complete state it wants persisted, including all four cricket innings (when applicable). Do not attempt to diff or merge server-side. Unknown `details.*` keys MUST still be preserved verbatim from the stored document.

```jsonc
Response 200: Match
Response 404: { "error": "Match not found" }
```

### `DELETE /matches/:slug`

```jsonc
Response 204
```

## Featured match

Exactly one match (or zero) is marked as featured at a time and surfaced on the public homepage. The backend MUST enforce this invariant atomically.

### `GET /matches/featured`

```jsonc
Response 200: Match | null                                                       // null when no match is currently featured
```

### `PUT /matches/:slug/featured`

Marks the given match as featured. The backend MUST atomically unset `featured` on any previously featured match in the same transaction, so at most one match is featured at any time.

```jsonc
Response 200: Match                                                              // the now-featured match (with `featured: true`)
Response 404: { "error": "Match not found" }
```

### `DELETE /matches/:slug/featured`

Clears the featured flag on the given match. No-op if it wasn't featured.

```jsonc
Response 204
```

## Bulk schedule import

### `POST /matches/upload-schedule`

Bulk-import match schedules from a CSV or Excel file. Each row becomes one match record. **Protected** — requires `x-access-token` header.

`Content-Type: multipart/form-data`

```
Request fields:
  tournamentSlug  string?   Optional. Associates all imported matches with this tournament.
  file            File      Required. Accepted formats: .csv, .xls, .xlsx
```

**CSV / Excel column order:**

| Column          | Type     | Required | Accepted values                                                                                              |
|-----------------|----------|----------|--------------------------------------------------------------------------------------------------------------|
| `homeTeamName`  | string   | yes      | –                                                                                                            |
| `awayTeamName`  | string   | yes      | –                                                                                                            |
| `sport`         | enum     | yes      | `CRICKET \| RUGBY \| SOCCER \| BADMINTON \| BOWLING \| MOTOR_SPORTS \| BASKETBALL \| NETBALL \| VOLLEYBALL \| CYCLING \| GOLF` |
| `matchType`     | enum     | no       | `LEAGUE \| KNOCKOUT \| FRIENDLY \| INDEPENDENT`                                                              |
| `scheduledAt`   | datetime | no       | ISO8601 or `YYYY-MM-DD HH:mm`                                                                                |
| `venue`         | string   | no       | –                                                                                                            |
| `status`        | enum     | no       | `SCHEDULED \| LIVE \| COMPLETED \| CANCELLED \| POSTPONED` (default: `SCHEDULED`)                            |
| `notes`         | string   | no       | –                                                                                                            |

The importer creates one `Match` per valid row. Rows that fail validation are reported in `errors[]` but do not abort the whole import unless **every** row fails (in which case the response is `422`).

```jsonc
Response 200:
{
  "imported": 12,
  "skipped":  1,
  "errors": [
    { "row": 3, "reason": "Missing homeTeamName" }
  ]
}

Response 400: { "error": "No file uploaded" }
Response 422: { "error": "All rows failed validation", "errors": [ ... ] }
```

## Status transitions

There is no state machine — the backend MUST accept any of the five `status` values on any update. Going `COMPLETED → LIVE` (e.g. correcting a premature completion from the cricket editor) is a legal admin action.

---

## Error format

All non-2xx responses use the project-wide error shape:

```json
{ "error": "human-readable message", "code": "OPTIONAL_MACHINE_CODE" }
```
